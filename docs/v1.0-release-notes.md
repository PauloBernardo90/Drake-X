# Drake-X v1.0 Release Notes

v1.0 lifts Drake-X from a strong per-sample static analysis platform
to a cross-sample, queryable, evidence-centric platform with external
evidence adapters, structured validation, platform-wide AI
auditability, first-class ELF workflow, multi-domain reporting, and a
real (if minimal) job/queue foundation.

## Platform capabilities added

### Cross-sample correlation + global graph query
- `drake_x/correlation/` — deterministic pairwise correlator over
  every persisted evidence graph in the workspace.
- Bases: shared imports, shared shellcode prefixes, shared indicator
  clusters, shared protection profiles, shared IOC values.
- `drake correlate run -w <ws>` surfaces pairwise matches with
  evidence-backed node IDs on both sides.
- `drake graph query -w <ws>` filters nodes across every session by
  kind, domain, label substring, data substring, or min confidence.

### External evidence ingestion
- `drake_x/integrations/ingest/` — adapter registry + JSON adapter
  (`drake ingest evidence <file> --type json -w <ws>`).
- `drake_x/models/external_evidence.py` — mandatory provenance
  (source tool, source file, ingested-at, adapter, trust, notes) on
  every imported record.
- Imported nodes carry `external: true` and preserve the provenance
  block under `node.data['provenance']`; they remain distinguishable
  from Drake-generated evidence in every downstream surface.

### Structured validation plan (multi-domain)
- `drake_x/models/validation_plan.py` — `ValidationPlan` +
  `ValidationItem` with status (planned / executed / confirmed /
  rejected), priority, hypothesis, rationale, suggested steps,
  expected evidence, suggested tool, domain, and graph node IDs for
  backing evidence.
- `drake_x/normalize/validation/planner.py` — deterministic planner
  that walks a session's graph and emits items from PE indicators,
  suspected shellcode, protection-interaction, and ingested
  findings.
- `drake_x/reporting/validation_writer.py` — Markdown writer.
- SQLite-backed persistence (`validation_plans` table).
- CLI: `drake validate plan`, `drake validate show`,
  `drake validate export`.

### Platform-wide AI auditability
- `drake_x/ai/audited_run.py` — single shared `run_audited(…)` helper
  used by the PE path and the generic `drake ai` tasks. Writes an
  audit record on success, on unreachable runtime, and on non-JSON
  responses.
- The PE exploit-assessment wiring was refactored through the shared
  helper; feature parity preserved, behaviour unchanged.

### First-class ELF workflow
- `drake_x/modules/elf_analyze.py`,
  `drake_x/integrations/binary/elf_parser.py`,
  `drake_x/normalize/binary/elf_normalize.py`,
  `drake_x/reporting/elf_report_writer.py`,
  `drake_x/models/elf.py`.
- CLI: `drake elf analyze <file>`.
- Covers header, sections, imports, protections (NX, PIE, RELRO,
  canary, FORTIFY), deterministic import-risk classification, graph
  ingestion. Exploit-awareness parity with PE is intentionally
  deferred beyond v1.0.

### Multi-domain case report
- `drake_x/models/case_report.py`,
  `drake_x/reporting/case_report_writer.py`.
- `drake report case -w <ws>` emits a consolidated Markdown or JSON
  document spanning every session's graph + correlations +
  validation plans, with explicit per-session domain tagging.
- Specialized per-domain writers remain the authoritative
  per-session document; this is an additional aggregation view.

### Distributed execution foundation (experimental)
- `drake_x/execution/` — `Job`, `JobStatus`, `LocalQueue`,
  `LocalWorker`, handler registry.
- SQLite-backed queue persists job history (attempts, started/
  finished timestamps, error) for audit.
- Handler abstraction mirrors what a future remote worker deployment
  would need. "No handler registered" is non-transient and the
  worker abandons immediately.
- **Status: experimental.** Interfaces may evolve before v1.1.

## CLI surface additions

```
drake correlate run   -w <workspace> [--min-shared N] [--format text|json]
drake graph query     -w <workspace> [--kind K] [--domain D] [--label S]
                                     [--data S] [--min-confidence F]
                                     [--format text|json]
drake ingest evidence <file> -w <workspace> [--type json]
                                            [--session ID] [--trust L]
drake ingest list-adapters
drake validate plan   <session> -w <workspace>
drake validate show   <session> -w <workspace> [--format text|json]
drake validate export <session> -w <workspace> -o <file>
drake elf analyze     <file> [-w <workspace>] [-o <output-dir>]
drake report case     -w <workspace> [-o <file>] [--format markdown|json]
```

## Compatibility

- `drake_x.__version__` is now `1.0.0`; `pyproject.toml` bumped; STIX
  `generator_version` reads the package version dynamically.
- PE CLI flags and output files unchanged.
- Audit log shape unchanged — the shared `run_audited` helper writes
  the same `AIAuditRecord` schema the PE path used.
- `Workspace` gained a `.storage` convenience property; existing code
  constructing `WorkspaceStorage(ws.db_path)` continues to work.
- No runtime dependencies added. ELF parsing uses `pyelftools` when
  present; degrades to header-only stdlib parsing otherwise.

## Known limitations

- ELF pipeline is surface-level; no exploit indicators, no shellcode
  carving, no protection-interaction assessment yet.
- Ingestion ships one concrete adapter (JSON). PCAP, ETW, and
  STIX-in adapters are future work — the registry is ready for them.
- Execution foundation is single-writer / single-worker local only.
  No RPC, no remote executors, no cluster scheduler.
- Correlation is O(N²) in session count. Fine for realistic
  workspaces; a signature-bucket join is the natural next optimization
  if a workspace grows into the thousands.
- Validation-plan *execution* is out of scope — plans are structured
  analyst checklists, not runnable commands. Drake-X does not execute
  plan items automatically.

## Beyond v1.0

- ELF exploit-awareness parity with PE.
- Additional ingest adapters (PCAP, ETW, STIX-in).
- Remote execution backends behind the existing `Queue` protocol.
- Campaign clustering on top of the correlator.
- AI audit schema versioning (`schema_version` field on
  `AIAuditRecord`).
